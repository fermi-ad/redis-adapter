# System Simulation Overview

Full-scale accelerator simulation using `adapters/docker-compose.yml` (system profiles). Each digitizer gets its own Redis 7.4 instance — the twin and adapter for that digitizer share it. Every device twin generates synthetic waveforms at 20 Hz; every adapter processes them and publishes derived results back to the same Redis.

Assumptions: **10,000 samples per channel**, **float32** (4 bytes), **20 Hz** trigger rate (50 ms interval), **trim=100** (5 seconds of history).

Each digitizer group: 1 Redis + 1 twin + 1 adapter (BPM), or N twins + N adapters sharing 1 Redis (BLM/BCM).

Generated by `adapters/generate_system.py`.

## Data Rates

All rates assume 20 Hz update (50 ms trigger interval) and `trim=100` (5 seconds of stream history per key).

|--------|-----------------|------------------|----------------------|
| BPM twin (8ch) | 625.0 KiB | 1 | 12.2 MiB/s |
| BPM twin (4ch) | 312.5 KiB | 1 | 6.1 MiB/s |
| BPM adapter (8ch) | 1,055 KiB | 32 | 20.6 MiB/s |
| BPM adapter (4ch) | 586 KiB | 20 | 11.4 MiB/s |
| BLM twin | 39.1 KiB | 1 | 0.8 MiB/s |
| BLM adapter | 78.1 KiB | 8 | 1.5 MiB/s |
| BCM twin | 312.5 KiB | 2 | 6.1 MiB/s |
| BCM adapter | 195.3 KiB | 29 | 3.8 MiB/s |

### Per-Redis data rates

Each Redis instance only handles the load from its digitizer:

| Redis group | Writers | XADD ops/s | Payload throughput |
|-------------|---------|------------|-------------------|
| `redis-bpm-*` (8ch) | 2 (twin + adapter) | 660 | 32.8 MiB/s |
| `redis-bpm-*` (4ch) | 2 | 420 | 17.5 MiB/s |
| `redis-blm-*` (full) | 16 (8 twins + 8 adapters) | 1,440 | 18.2 MiB/s |
| `redis-bcm-*` | 8 (4 twins + 4 adapters) | 2,480 | 39.6 MiB/s |

All per-Redis rates are well within single-instance capacity (Redis 7.4 handles ~250k ops/s over UDS, see [BENCHMARKS.md](BENCHMARKS.md)).

## Redis Memory Footprint

With `trim=100` (5 seconds at 20 Hz), each stream retains 100 entries.

### Per-Redis memory (trim=100)

| Redis group | Streams | Payload per stream (×100) | Total payload | Est. Redis memory |
|-------------|---------|--------------------------|---------------|-------------------|
| `redis-bpm-*` (8ch, ×55) | 33 | 26 wf × 3.8 MiB + 2 fft × 1.9 MiB + 4 scalar ≈ 0 | ~103 MiB | ~160 MiB |
| `redis-bpm-*` (4ch, ×15) | 21 | 14 wf × 3.8 MiB + 2 fft × 1.9 MiB + 4 scalar | ~57 MiB | ~90 MiB |
| `redis-blm-*` (full, ×16) | 72 | 24 wf × 3.8 MiB + 48 scalar | ~92 MiB | ~140 MiB |
| `redis-blm-17` (remainder) | 18 | 6 wf × 3.8 MiB + 12 scalar | ~23 MiB | ~35 MiB |
| `redis-bcm-*` (×5) | 124 | 20 wf × 3.8 MiB + 4 filt × 3.8 MiB + 96 scalar | ~92 MiB | ~140 MiB |

### System-wide memory totals (trim=100)

| Component | Redis instances | Payload total | Est. Redis memory total |
|-----------|----------------|---------------|------------------------|
| BPM (8ch) | 55 | 5.5 GiB | 8.6 GiB |
| BPM (4ch) | 15 | 0.8 GiB | 1.3 GiB |
| BLM | 17 | 1.5 GiB | 2.3 GiB |
| BCM | 5 | 0.4 GiB | 0.7 GiB |
| Redis process overhead | 92 × ~10 MiB | — | 0.9 GiB |
| **Total** | **92** | **~8.3 GiB** | **~13.8 GiB** |

Redis overhead factor is ~1.6x for stream entries with large blob fields. Each Redis process also has ~10 MiB of base memory overhead. Actual memory depends on Redis version and `jemalloc` behavior. Monitor with `INFO memory`.

## XADD Operation Budget

| Metric | Value |
|--------|-------|
| Total XADD ops/s (all Redis) | 78,400 |
| Max per-Redis XADD ops/s | ~2,480 (BCM digitizer) |
| Min per-Redis XADD ops/s | ~420 (4ch BPM digitizer) |

With 1 Redis per digitizer, the load is distributed across 92 independent instances. The busiest Redis (BCM, serving 4 devices) sees ~2,480 ops/s and ~39.6 MiB/s — far below the ~250k ops/s and ~1.3 GiB/s capacity of a single Redis 7.4 instance.

## Docker Compose

All profiles live in a single `adapters/docker-compose.yml`, generated by `adapters/generate_system.py`.

| Profile | Description | Services |
|---------|-------------|----------|
| `pipeline` | Individual signal-processing adapters | 9 |
| `bpm-demo` | BPM composite adapter + twin | 3 |
| `blm-demo` | BLM adapter + twin | 3 |
| `bcm-demo` | BCM adapter + twin | 3 |
| `tiny`..`full` | System simulation (see profile table above) | 64–532 |
| `tui` | Instrument TUI and Redis TUI (interactive) | 2 |

### Running the full system

```bash
cd adapters

# Generate configs (if not already present)
python3 generate_system.py

# Build all images (shared Dockerfile, built once)
docker compose --profile full build

# Start all 532 services
docker compose --profile full up -d

# Monitor
docker compose --profile full ps
docker compose --profile full exec redis-bpm-001 redis-cli INFO memory
docker compose --profile full exec redis-blm-01 redis-cli DBSIZE

# TUI (interactive)
docker compose --profile tui run --rm inst-tui /inst-tui --config /etc/inst-tui/inst-tui-hosts-full.txt

# Stop
docker compose --profile full down
```
